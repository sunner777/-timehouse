## 核心教训（绝对禁止再犯）

### 1. 禁止使用国外占位图服务
**问题**：选择了via.placeholder.com等国外服务，导致中国大陆用户无法访问
**后果**：图片始终无法显示，浪费大量排查时间
**正确做法**：
- 必须使用国内可访问的服务或真实云存储
- 本项目使用火山引擎TOS作为真实云存储

### 2. 禁止在前端写死虚假数据
**问题**：上传和删除功能在前端写死了虚假数据（模拟数据）
**后果**：一直找不到图片显示不出的真正原因，以为是后端或网络问题
**正确做法**：
- 所有数据操作必须走真实API
- 前端只负责调用API和服务，不存储业务数据
- 调试时先确认API返回真实数据

### 3. 正确使用SDK导入方式
**问题**：火山引擎TOS SDK导入方式错误，使用了旧版写法
**后果**：代码无法运行，浪费时间排查
**正确做法**：
- 先了解最新的SDK导入方式再写代码
- 查看官方文档或SDK源码，确认正确的导入和使用方式
- 对于新版本SDK，注意API方法名可能发生变化

### 4. 核心原则
**原则1**：前后端数据必须一致
- 前端显示的数据必须来自后端API
- 禁止在前端硬编码模拟业务数据

**原则2**：云存储必须使用国内服务
- 国内用户访问国外服务受限
- 选择阿里云OSS、腾讯云COS、火山引擎TOS等国内云存储

**原则3**：私有桶访问必须使用签名URL
- 私有桶中的对象无法直接公开访问
- 必须通过预签名URL（带签名参数的临时访问链接）访问
- 签名URL有过期时间，确保安全性

**原则4**：技术方案必须考虑中国大陆用户访问最佳性能
- 所有服务选择国内服务商
- 选择靠近用户的区域部署服务
- 避免使用可能被网络限制的服务

**原则5**：遇到相同错误多次修改无效时的处理思路
- 重新分析错误信息，可能理解有误
- 检查是否有沙箱环境限制
- 考虑是否需要用户在沙箱外帮助修复（如网络配置、权限设置等）
- 及时向用户说明情况，寻求必要的协助

### 照片接口
- POST /api/v1/photos/upload - 上传照片
- GET /api/v1/photos - 获取照片列表
- GET /api/v1/photos/:id - 获取照片详情
- PUT /api/v1/photos/:id - 更新照片信息
- DELETE /api/v1/photos/:id - 删除照片
- POST /api/v1/photos/batch-delete - 批量删除
- POST /api/v1/photos/tos-upload-signature - 获取TOS上传签名

## TOS集成流程

### 上传流程
1. 前端请求后端获取TOS上传签名
2. 后端生成预签名URL并返回给前端
3. 前端直接通过预签名URL上传文件到TOS
4. 上传成功后，前端通知后端保存照片元数据

### 下载流程
1. 前端请求获取照片列表
2. 后端从数据库获取照片元数据
3. 后端为每个照片生成下载预签名URL
4. 前端使用带签名的URL显示图片

## 常见问题排查

### 图片不显示
1. 检查URL格式是否正确（无多余空格、反引号等）
2. 检查是否为私有桶（需要签名URL）
3. 检查签名URL是否过期
4. 检查前端是否正确处理URL

### 上传失败
1. 检查TOS配置是否正确
2. 检查签名生成是否正确
3. 检查CORS配置是否允许跨域
4. 检查桶权限设置

### 后端无法启动
1. 检查端口是否被占用
2. 检查数据库连接配置
3. 检查环境变量文件(.env)
4. 检查依赖是否完整安装

## V2.0 开发经验（2026-05-16）

### Flutter 导航与状态

**原则6：导航用 context.push() 而非 context.go()**
- go() 替换整个路由栈，会破坏底部导航栏的选中状态
- push() 压入新页面，pop() 自然回到上一页
- 对于二级页面（大图浏览、成员管理），用 push，顶部 AppBar 默认返回箭头自动调用 pop()

**原则7：对话框用 showModalBottomSheet 而非 Scaffold.bottomSheet**
- Scaffold.bottomSheet 设为 null 不会关闭已打开的 Sheet
- showModalBottomSheet 每次都是全新实例，通过 Navigator.pop(value) 返回结果

**原则8：跨组件共享的缓存数据放在 Provider 里**
- 多个 Widget 用到同一份数据时，缓存放 Provider 而非某个 Widget 的本地 state
- 配合 notifyListeners + context.watch，数据变更自动传播到所有监听者
- 避免"数据在 A 组件改了但 B 组件不知道"的问题

**原则9：await context.push() 模式用于父子页面数据联动**
- A 页面 push 到 B 页面，用 await 等待 B pop 返回
- B 操作完 pop，A 的 await 恢复执行，在 then 中刷新数据
- 比传递回调函数更简洁直观

**原则10：独立路由的页面不应假设数据已预加载**
- 大图浏览页可能在任何时机进入，不能假设 photoProvider.photos 已填充
- 应支持按 id 从 API 单独加载（API 兜底），本地缓存命中则直接使用

### 后端权限

**原则11：共享组场景下照片查询需放开 userId 过滤**
- 旧逻辑：Photo.findById 按 userId + photoId 查
- 共享组场景：其他成员上传的照片 userId 不匹配 → 404
- 正确做法：先按 userId 查（本人照片），未命中则按 photoId 查 + family 成员身份校验
