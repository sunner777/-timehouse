### 开发经验总结

1. **showModalBottomSheet 而非 Scaffold.bottomSheet** — 后者设 null 不会自动关闭已打开的 Sheet，导致 UI 无响应。对话框类交互统一用 showModalBottomSheet，通过 Navigator.pop(value) 返回结果。

2. **context.push() 而非 context.go()** — go() 替换整个路由栈，破坏底部导航选中状态；push() 压栈，pop() 自然回到上一页，导航状态保持一致。

3. **跨组件共享缓存放在 Provider 里** — 多个 Widget 用到同一份数据时，缓存放在 Provider 而非某个 Widget 的本地 state。配合 notifyListeners + context.watch，数据变更自动传播到所有监听者。

4. **共享组照片访问需后端放开 userId 过滤** — Photo.findById 加了 userId 条件后，其他成员上传的照片查不到。需要 findByIdOnly + family 成员身份校验两步走。

5. **await context.push() 模式** — 从 A 页面 push 到 B，B 操作完 pop 回来，A 用 await 等待 push 完成，然后在 then 里刷新数据。比传递回调更简洁。

6. **API 兜底加载** — 不要假设 photoProvider.photos 总是已填充。大图浏览页作为独立路由，可能在列表页请求之前就进入，应支持从 API 按 id 单独加载。

7. **后端响应格式要保持一致** — getPhotos 返回 `{ photos: [...] }`，getPhotoDetail 返回直接是 photo 对象。前端解析时用 `data['photo'] ?? data` 兼容两种格式。

8. **FutureBuilder 的 future 要稳定** — 每次 build 创建新 Future 会导致无限循环。解决方式：用带缓存的 Provider 方法，缓存命中时同步返回，FutureBuilder 看到已完成 Future 不重复请求。

### 今日教训（重要）
1. **正确使用SDK导入方式**：TOS SDK新版必须使用解构导入`const { TosClient } = require('@volcengine/tos-sdk')`，而非旧版`const TOS = require('@volcengine/tos-sdk')`
2. **技术方案必须考虑中国大陆用户**：所有服务选择国内服务商，避免使用可能被网络限制的服务
3. **多次修改无效时寻求帮助**：如果遇到相同错误多次修改无效，需要思考是否有沙箱环境限制，及时向用户说明情况
4. **Provider必须正确注册**：使用Provider时必须在main.dart的MultiProvider中注册，否则会导致红屏崩溃
5. **FutureBuilder慎用**：FutureBuilder在每次Widget构建时会重新执行future，可能导致无限循环，必要时使用缓存机制

### Bug 修复：换设备登录后共享组消失

#### 问题
用户 15306814275 在另一台手机上登录后，创建的共享组全部不显示。

#### 排查
- ✅ 生产数据库数据完好（用户 3 拥有 2 个共享组）
- ✅ 后端 API 正常（本地和 HTTPS 均返回正确数据）
- 🐛 **根因：前端 `families_screen.dart` 的 `build` 方法有静默错误吞没**
  - 当 `getFamilies()` API 调用失败时（网络波动、解析异常等），`_errorMessage` 被设置但 UI 完全不显示
  - UI 只判断 `isLoading` → 空状态，导致 API 异常时显示「还没有家人共享」让用户误以为数据丢失

#### 修复（commit `613fa67`）
1. **`families_screen.dart`** — 新增 `_buildErrorState()`，错误状态优先于空状态显示，展示错误消息 + 重试按钮
2. **`family_provider.dart`** — `getFamilies()` 改为逐个解析 JSON，单项失败不阻塞其余数据
3. **`my_space_screen.dart`** — 同样增加错误态处理

#### 待用户验证
- 需要重新构建 APK 并在另一台手机测试
- 如果问题依然存在，错误消息会直接显示在界面上，方便进一步定位
