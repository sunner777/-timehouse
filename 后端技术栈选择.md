# 时光家项目 - 后端技术栈选择

## 1. 核心技术栈

| 组件 | 技术选型 | 版本 | 推荐理由 | 兼容性说明 |
|------|----------|------|----------|------------|
| **运行时** | Node.js | 18.17.0 LTS | 适合I/O密集型应用，处理照片上传和存储操作性能优异 | 推荐使用18.17.0 LTS版本，避免使用非LTS版本导致的稳定性问题 |
| **Web框架** | Express.js | 4.18.2 | 轻量级、灵活，易于扩展，适合构建RESTful API | 与Node.js 18.x完全兼容 |
| **关系型数据库** | MySQL | 8.0.34 | 适合存储用户、家庭组、权限等结构化数据，事务支持完善 | 推荐使用8.0.34版本，避免8.0.28以下版本的性能问题 |
| **文档型数据库** | MongoDB | 6.0.12 | 适合存储照片元数据等半结构化数据，查询灵活 | 6.0.x版本稳定性较好，避免使用5.0.x版本的兼容性问题 |
| **缓存** | Redis | 7.0.15 | 用于会话存储、接口缓存、限流，提升性能 | 7.0+版本支持更多新特性，性能更优 |
| **认证** | JWT | jsonwebtoken 9.0.2 | 无状态认证，便于水平扩展 | 与Node.js 18.x完全兼容 |

### 1.1 依赖版本锁定策略
- 使用 `package-lock.json` 锁定依赖版本，确保团队协作时依赖一致性
- 定期更新依赖包，保持安全性和性能
- 建立依赖升级测试流程，避免版本更新引入的问题

## 4. 技术栈优势

1. **性能优异**：Node.js在处理并发请求和I/O操作时表现出色，适合照片上传和存储场景
2. **开发效率高**：JavaScript全栈开发，前后端代码风格一致，开发周期短
3. **生态成熟**：丰富的npm包，社区活跃，问题解决快速
4. **易于扩展**：模块化设计，支持微服务架构
5. **与火山引擎集成**：通过SDK轻松集成TOS、CDN等服务
6. **安全可靠**：完善的认证机制和权限控制

## 5. 部署方案

- **开发环境**：本地Docker容器
- **测试环境**：火山引擎容器服务
- **生产环境**：
  - API服务：火山引擎SCF + API网关
  - 数据库：
    - 火山引擎RDS MySQL（高可用版）：主从架构，自动故障切换
    - 火山引擎MongoDB（副本集）：3节点副本集，高可用
  - 缓存：
    - 火山引擎Redis（主从架构）：持久化存储，防止数据丢失
  - 存储：
    - 火山引擎TOS + CDN：加速访问
    - 跨区域备份：重要数据多区域存储

### 5.1 高可用与容灾策略
- **MySQL高可用**：RDS主从架构，自动故障切换，RPO=0，RTO<30秒
- **MongoDB高可用**：3节点副本集，自动选举，数据冗余
- **Redis高可用**：主从架构，持久化开启，定期备份
- **数据备份**：
  - MySQL：每日全量备份，每小时增量备份
  - MongoDB：每日全量备份，实时 oplog 备份
  - TOS：跨区域复制，防止单区域故障
- **容灾演练**：定期进行故障演练，确保灾备方案有效

## 7. 核心功能实现

### 7.1 认证系统
- JWT token生成与验证
- 密码加密存储
- 权限控制中间件

### 7.2 照片管理
- **临时上传签名生成**：后端生成TOS临时上传签名，前端直接上传
- **上传方式**：前端直传TOS，避免后端中转，提高上传速度
- **大文件分片上传**：支持5M以上大文件分片上传，提升稳定性
- **文件校验**：使用MD5校验防篡改，确保文件完整性
- **照片元数据管理**：存储照片的拍摄时间、位置、尺寸等信息
- **批量操作支持**：支持批量上传、删除、分享等操作

### 7.3 家庭组管理
- **组创建与管理**：支持创建、修改、删除家庭组
- **成员邀请与管理**：通过邀请码或手机号邀请成员，支持移除成员
- **权限控制**：
  - 细粒度权限：查看/编辑/删除/管理
  - 基于角色的权限模型：管理员/成员/访客
- **组内照片共享**：
  - 组内成员可查看所有共享照片
  - 权限控制下的操作权限
  - 实时同步机制：Redis缓存刷新，确保权限变更及时生效

### 7.4 人脸识别
- **识别接口集成**：集成火山引擎AI服务进行人脸识别
- **识别结果存储**：将识别结果存储在TOS和MongoDB中
- **缓存策略**：
  - 高频查询的人物分类数据缓存到Redis
  - 识别结果缓存，避免重复识别
- **失败重试机制**：
  - 异步识别任务失败自动重试
  - 最大重试次数3次，指数退避
- **跨设备一致性**：
  - 识别结果以用户/家庭组为维度同步
  - 多设备之间保持一致的人物分类
- **人物分类管理**：支持手动修改人物名称，合并相似人物

## 8. 安全性考虑

- **HTTPS传输**：所有API请求使用HTTPS
- **API签名**：防止请求篡改
- **限流防护**：防止恶意请求
- **参数校验**：输入数据验证
- **TOS访问控制**：最小权限原则

## 9. 性能优化

- **缓存策略**：Redis缓存热点数据
- **异步处理**：照片上传和处理异步化
- **数据库索引**：优化查询性能
- **CDN加速**：提升照片访问速度
- **连接池**：数据库连接复用

## 10. 监控与可观测性

- **核心指标监控**：
  - API响应耗时：P95/P99指标
  - 照片上传成功率：实时监控
  - Redis缓存命中率：优化缓存策略
  - 数据库慢查询：定期分析
  - 服务错误率：及时发现异常
- **日志收集**：
  - 接入ELK栈：集中管理日志
  - 结构化日志：便于分析
  - 错误日志告警：及时响应
- **告警机制**：
  - SCF执行失败告警
  - TOS存储量超标告警
  - 数据库连接异常告警
  - API错误率超阈值告警
- **监控平台**：
  - 火山引擎监控服务
  - 自定义仪表盘：关键指标可视化

## 11. 技术风险与兜底预案

- **SCF资源不足**：
  - 兜底方案：容器化部署到火山引擎容器服务
  - 自动扩容：根据流量自动调整容器实例数
- **MongoDB性能下降**：
  - 索引优化：定期分析查询计划，添加合适索引
  - 数据分片：当数据量达到阈值时，进行水平分片
  - 读写分离：读操作路由从节点
- **Node.js性能瓶颈**：
  - 人脸识别处理：使用独立的服务进程处理CPU密集型任务
  - 事件循环优化：避免在主进程中执行耗时操作
  - 集群模式：启用Node.js集群，充分利用多核CPU
- **TOS服务异常**：
  - 备用存储：配置备用对象存储服务
  - 本地缓存：临时缓存关键照片，保证用户体验
- **网络延迟**：
  - 多区域部署：靠近用户的区域部署服务
  - 边缘节点：利用CDN边缘节点缓存静态资源

## 12. 总结

此技术栈选择方案既满足时光家项目的功能需求，又考虑了性能、安全性和可扩展性，为项目的成功实施提供了可靠的技术基础。通过完善的版本兼容性管理、核心场景的技术落地细节、高可用与容灾策略、监控与可观测性体系以及技术风险的兜底预案，确保了项目的稳定运行和可持续发展。
